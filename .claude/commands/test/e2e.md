# E2E 테스트 — Playwright MCP

ultrathink

## 테스트 대상 결정

$ARGUMENTS가 주어지면:

- 마일스톤 번호(M1~M7)이면: 해당 마일스톤의 구현 범위
- feature 이름이면: 해당 feature의 구현 범위

$ARGUMENTS가 없으면:

- `docs/ROADMAP.md`를 읽어 현재 **완료** 상태인 마일스톤을 파악
- 완료된 마일스톤의 구현사항을 E2E 테스트 대상으로 선정

## 사전 준비

1. `docs/ROADMAP.md`를 읽어 대상 마일스톤의 구현 범위를 파악한다
2. `docs/PRD.md`를 읽어 해당 기능의 상세 요구사항을 파악한다
3. 개발 서버가 실행 중인지 확인하고, 실행 중이 아니면 사용자에게 `pnpm dev`를 실행하라고 안내한다

## E2E 테스트 시나리오 작성 및 실행

Playwright MCP 도구를 사용하여 실제 브라우저에서 테스트합니다.

### 마일스톤별 테스트 시나리오

#### M1: 공통 인프라

- Toast 시스템이 화면에 표시되는지 확인
- 레이아웃(Header, Navigation)이 정상 렌더링되는지 확인

#### M2: 회원 기능

- **회원가입 플로우**: `/signup` → 필수 필드 입력 → 실시간 유효성 검증 → 가입 완료 → 로그인 페이지 이동 + Toast
- **로그인 플로우**: `/login` → 이메일/비밀번호 입력 → 로그인 성공 → `/videos` 이동
- **로그인 실패**: 미가입 이메일, 비밀번호 불일치 에러 메시지 표시
- **회원정보 수정**: `/profile` → 기존 정보 로드 → 수정 → 저장 + Toast
- **인증 가드**: 비로그인 상태에서 `/profile` 접근 → `/login` 리다이렉트
- **로그아웃**: 로그아웃 버튼 → 세션 삭제 → 로그인 페이지 이동

#### M3: 동영상 리스트/상세

- **리스트 조회**: `/videos` → 영상 목록 렌더링 (제목, 업로더, 업로드일)
- **페이지네이션**: 페이지 이동 → 다른 영상 목록 표시
- **상세 페이지**: 영상 클릭 → `/videos/[id]` → 상세 정보 표시 (제목, 업로더, 설명)
- **목록 돌아가기**: "목록으로 돌아가기" 버튼 → `/videos` 이동
- **에러 처리**: API 실패 시 에러 UI 표시

#### M4: 커스텀 비디오 플레이어

- **재생 컨트롤**: 재생/일시정지 버튼 동작
- **시간 표시**: `m:ss / m:ss` 포맷 표시
- **프로그래스바**: 클릭/드래그로 시간 이동
- **재생 속도**: 속도 변경 UI 동작
- **키보드 단축키**: Space, 좌우 화살표, F, ESC
- **컨트롤 숨김**: 마우스 이동 시 표시, 3초 유휴 시 숨김

#### M5: 재생 정책 및 메시지

- **영상 종료**: 끝까지 재생 → 일시정지, 재생 클릭 → 처음부터
- **90% 시청 완료**: 모달 표시 → 닫기 → 리뷰 모달
- **리뷰 제출**: 평점 입력 → 제출 → localStorage 저장

#### M6: 영상 노트

- **노트 CRUD**: 생성, 조회, 수정, 삭제
- **노트 검색**: 텍스트 필터링
- **시점 이동**: 노트 클릭 → 해당 시점으로 영상 이동
- **비로그인 안내**: 비로그인 시 "로그인 후 이용 가능" 표시

## 테스트 실행 방법

Playwright MCP 도구를 활용합니다:

1. `mcp__playwright__browser_navigate`로 페이지 이동
2. `mcp__playwright__browser_snapshot`으로 현재 페이지 상태 캡처
3. `mcp__playwright__browser_click`으로 요소 클릭
4. `mcp__playwright__browser_fill_form`으로 폼 입력
5. `mcp__playwright__browser_press_key`로 키보드 입력
6. `mcp__playwright__browser_wait_for`로 비동기 요소 대기
7. `mcp__playwright__browser_take_screenshot`으로 스크린샷 촬영

## 테스트 코드 검증

E2E 테스트 후, 기존 유닛/통합 테스트 코드도 검증합니다:

1. `pnpm exec vitest run` 전체 테스트 실행
2. 실패하는 테스트가 있으면 원인 분석
3. 테스트가 실제 요구사항을 올바르게 검증하는지 확인:
   - 테스트가 구현이 아닌 **동작**을 검증하는가
   - 엣지 케이스가 커버되어 있는가
   - 모킹이 과도하지 않은가
   - 테스트 설명이 명확한가

## 주의사항

- 개발 서버가 반드시 실행 중이어야 한다 (http://localhost:3000)
- localStorage 상태에 의존하는 테스트는 각 시나리오 시작 전 초기화를 고려한다
- 네트워크 요청이 포함된 테스트는 적절한 대기 시간을 설정한다
- 스크린샷은 실패한 시나리오에서만 촬영한다
