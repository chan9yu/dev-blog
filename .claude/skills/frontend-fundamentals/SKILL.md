# Frontend Fundamentals: 코드 품질

좋은 프론트엔드 코드는 **변경하기 쉬운** 코드다. 가독성, 예측 가능성, 응집도, 결합도 네 가지 원칙으로 코드 품질을 판단한다. 이 원칙들은 때로 충돌하므로 상황에 따라 우선순위를 정해야 한다.

각 원칙의 상세 코드 예제는 `references/examples.md` 참조.

## 원칙 1: 가독성 (Readability)

읽는 사람이 한 번에 고려해야 하는 맥락을 최소화한다 (인간의 단기 기억 ≈ 6-7개).

### 맥락 줄이기

- **동시에 실행되지 않는 코드를 분리한다.** 조건에 따라 완전히 다른 UI를 보여주는 컴포넌트는 분기를 섞지 말고, 각 분기를 별도 컴포넌트로 분리한다.
- **구현 세부사항을 추상화한다.** 저수준 로직이 컴포넌트의 목적을 가리면, 잘 이름 붙인 래퍼(HOC, Guard 컴포넌트, 커스텀 훅)로 추출하여 부모 컴포넌트가 높은 수준의 설명처럼 읽히게 한다.
- **로직 유형이 아닌 관심사별로 훅/함수를 분리한다.** 페이지의 모든 쿼리 파라미터를 관리하는 `usePageState()` 훅은 `useXxxParam()` 형태의 개별 훅으로 분리한다 — 각각 명확한 이름과 좁은 책임을 갖도록.

### 이름 잘 짓기

- **복잡한 조건에 이름을 붙인다.** 인라인 `a && b || c` 대신 `isSameCategory`, `isPriceInRange` 같은 이름 있는 불리언으로 의도를 드러낸다.
- **매직 넘버에 이름을 붙인다.** `delay(300)` → `delay(ANIMATION_DELAY_MS)` — 이름이 그 값이 _왜_ 존재하는지 알려준다.

### 위에서 아래로 읽히게

- **시선 이동을 최소화한다.** 한 줄을 이해하기 위해 먼 함수로 점프했다 돌아와야 한다면, 인라인하거나 구조를 바꿔 독자가 아래로 자연스럽게 흐를 수 있게 한다.
- **깊은 삼항 중첩을 피한다.** 중첩된 `? : ? :` 체인은 early return을 쓰는 IIFE나 `if/else if`로 대체한다.
- **비교 연산을 왼쪽에서 오른쪽으로 쓴다.** `minPrice <= price && price <= maxPrice`는 수학처럼 읽히지만, `price >= minPrice && price <= maxPrice`는 그렇지 않다.

## 원칙 2: 예측 가능성 (Predictability)

동료가 이름과 시그니처만 보고 동작을 예측할 수 있는 코드를 작성한다.

- **중복되는 이름을 피한다.** 라이브러리가 `http`를 export하는데 래핑 모듈도 `http`로 이름 짓지 말 것 — `httpService`에 `getWithAuth` 같은 메서드명으로 추가된 동작을 드러낸다.
- **유사 함수의 반환 타입을 통일한다.** `useUser()`가 TanStack Query 객체를 반환하는데 `useServerTime()`이 raw 데이터만 반환하면, 동료는 매번 반환 타입을 확인해야 한다. 하나의 패턴을 정해 일관되게 따른다. Validation 함수도 마찬가지 — 어떤 건 boolean, 어떤 건 객체를 반환하지 말고, 모두 `{ ok, reason? }` 형태로 통일한다.
- **사이드 이펙트를 숨기지 않는다.** `fetchBalance`라는 함수가 몰래 로깅까지 한다면, 호출자는 로깅이 언제 발생하는지 예측할 수 없다. 함수를 시그니처에 정직하게 유지하고, 사이드 이펙트는 호출 측에서 명시적으로 조합한다.

## 원칙 3: 응집도 (Cohesion)

함께 변경되는 코드는 함께 위치한다.

- **함께 수정되는 파일을 같은 디렉토리에 둔다.** 타입별 디렉토리(`components/` 100개 파일)보다 기능별 디렉토리(`features/auth/components/`). 다른 도메인의 import가 `../../../video/hooks/useFoo`처럼 보이면 명확한 코드 스멜이다.
- **응집도를 위해 매직 넘버를 제거한다.** CSS 트랜지션과 JS delay가 같은 애니메이션 시간 `300`을 공유한다면, 상수로 추출해서 항상 함께 업데이트되게 한다.
- **폼의 응집도 수준을 고려한다.** 필드 수준 응집도(각 필드가 독립적으로 검증) vs 폼 수준 응집도(단일 스키마가 전체 검증). 필드가 독립적으로 변경되는지, 그룹으로 변경되는지에 따라 선택한다.

## 원칙 4: 결합도 (Coupling)

변경의 영향 범위를 제한한다.

- **책임을 개별적으로 관리한다.** "페이지의 모든 쿼리 파라미터를 관리"하는 훅은 모든 소비자를 모든 파라미터에 결합시킨다. 관심사별 훅으로 분리한다.
- **추상화가 결합도를 높인다면 코드 중복을 허용한다.** 두 페이지가 지금은 비슷한 바텀시트 로직을 공유하지만 나중에 달라질 수 있다면, 10줄 중복이 복잡한 설정이 필요한 공유 훅보다 낫다. 공유 훅은 변경할 때마다 모든 소비자를 테스트해야 하는 부담이 생긴다.
- **Composition 패턴으로 Props Drilling을 제거한다.** `keyword`, `items`, `onConfirm`을 3단계 래퍼 컴포넌트에 전달하는 대신, 데이터가 있는 곳에서 직접 자식을 렌더링하여 트리를 평탄화한다.

## 원칙이 충돌할 때

네 원칙은 서로 반대 방향으로 끌어당길 수 있다:

| 긴장 관계        | 상황                                                     | 해결 기준                                                              |
| ---------------- | -------------------------------------------------------- | ---------------------------------------------------------------------- |
| 가독성 vs 응집도 | 함께 변경되도록 추상화하면 읽기 어려워짐                 | 함께 수정하지 않으면 버그 발생 → 응집도 우선. 위험 낮음 → 가독성 우선. |
| 응집도 vs 결합도 | 중복 코드를 중앙화하면 응집도는 높아지나 소비자가 결합됨 | 동작이 항상 동일할 것 → 중앙화. 달라질 가능성 있음 → 중복 허용.        |

보편적인 정답은 없다. 코드의 생애 동안 무엇이 변경을 가장 쉽게 만들지 생각한다.
